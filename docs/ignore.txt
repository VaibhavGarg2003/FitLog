PROBLEMS ENCOUNTERED STEP 1

The architecture, tech stack, and workflow you’ve designed are legitimately **98% perfect**. This is a highly robust, modern SaaS architecture. The strict separation of concerns (Browser → API → Service → Repository → DB), the defense-in-depth security, and the LLM fallback chain are exactly the kind of architectural patterns that senior engineers expect to see in high-level system design discussions.

However, because you have a strict **"free of cost for development, deployment, and production"** constraint, there are three hidden bottlenecks in your current plan that will break your app once real users start hitting it.

Here is the 2% that needs tweaking to ensure it actually stays free and doesn't crash in production.

### 1. The Upstash Redis Trap (10,000 limit)

Your plan uses Upstash Redis for rate limiting and query caching. The free tier gives you **10,000 commands per day**.

* **The Problem:** If you put the rate limiter on *every* API route, one user session (logging in, opening the dashboard, fetching a workout, logging 5 sets) could easily trigger 30 API calls. With just 100 active users, you will hit 3,000 commands. If you also use Redis to cache dashboard queries, you will blow past the 10,000 limit by dinnertime, and Upstash will start rejecting requests, breaking the app.
* **The Fix:** **Do not rate-limit everything.** Only apply Upstash rate limiting to expensive/abusable routes: Auth endpoints, LLM endpoints (`/api/meal/parse`), and heavy mutation endpoints. For standard GET requests (like fetching workout history), rely on Next.js's built-in `fetch` cache or TanStack Query. Keep Redis strictly for protecting your LLM API keys.

### 2. The Vercel 10-Second Timeout vs. LLM Fallbacks

You are hosting on Vercel’s free Hobby tier, which has a **hard 10-second execution limit** for API routes.

* **The Problem:** Your LLM fallback chain is: `Gemini Flash → Groq → OpenRouter`. If Gemini is struggling and takes 8 seconds before failing, and then Groq takes 3 seconds, your total time is 11 seconds. Vercel will forcefully kill the request at 10 seconds, returning a 504 Gateway Timeout. The user gets nothing, and the fallback chain becomes useless.
* **The Fix:** Implement strict `AbortController` timeouts in your LLM router. Give Gemini a maximum of **4 seconds** to respond. If it doesn't, kill it and move to Groq (which is lightning-fast). You must ensure the *entire* chain resolves within ~8 seconds to give Prisma enough time to save the data and return the response before Vercel's 10-second execution axe falls.

### 3. Supabase "7-Day Pause" Automation

You noted in your context file that Supabase pauses free projects after 7 days of inactivity. You mentioned keeping a "cron ping active."

* **The Fix:** Don't rely on yourself to remember to visit the site. Since you already have GitHub Actions set up for CI/CD, create a simple `.github/workflows/keepalive.yml` that runs a `curl` command against a public `/api/health` route in your app every 3 days. This guarantees Supabase never goes to sleep, costing you $0.

### 4. Vercel Commercial Use Policy

This is a legal/TOS issue, not a technical one. Vercel’s Hobby (free) tier explicitly forbids commercial usage.

* **The Reality Check:** As long as the site is free to use, has no ads, no affiliate links, and no paywalls, you are perfectly fine on the Hobby tier. If you ever decide to monetize FitLog in the future, Vercel will require you to upgrade to the Pro tier ($20/mo).

### Summary Verdict

You don't need to rewrite any of your architectural documentation, but you *must* apply these constraints when we actually write the code for Step 2 and Step 3.

The foundation is absolutely rock solid. Handling these specific edge cases—resource starvation, cascading timeouts, and free-tier orchestration—is what separates a toy project from a production-grade system.

## Problem 1 — Double Counting Exercise Calories
### What is it saying?

Right now, when a user picks `MODERATE` activity level in the onboarding wizard, the TDEE formula multiplies their BMR by $1.55$. That $1.55$ already **assumes** they go to the gym 3-5 days a week. That assumption is baked permanently into their daily calorie target.

Now, in Step 3 of the build, when we build the workout logger, there will be a separate module that calculates calories burned during a workout session (using [cardio.ts](file:///c:/Fitness_app/lib/engine/cardio.ts) and [strength.ts](file:///c:/Fitness_app/lib/engine/strength.ts)).

The dangerous question is: **what do we do with that workout calorie number?**

```
Example: Vaibhav's TDEE = 2,695 kcal (already includes gym 4x/week)

He logs a gym session → cardio.ts calculates he burned 350 kcal.

❌ WRONG: Target becomes 2,695 + 350 = 3,045 kcal
   (Gym already counted in the 1.55 multiplier → counted TWICE)

✅ RIGHT: Keep target at 2,695 kcal.
   Show the 350 kcal as INFORMATION ONLY ("you burned ~350 kcal today")
   but don't add it to the daily budget.
```

This is exactly how MyFitnessPal gets it wrong for millions of users. They use a sedentary TDEE but then add workout calories on top, which is a legitimate model. But they don't warn you that you should pick `SEDENTARY` in settings, not `ACTIVE`, to avoid double-counting. Many people pick their true activity level, activate the exercise add-back, and end up eating way more than intended.

### Is this important?
**Yes. Critical. This is the most important point.** If we add session calories on top of a MODERATE TDEE, a user could silently overeat by 300-500 kcal every day and wonder why they are gaining fat instead of losing it. We need to decide our model before building the workout logger in Step 3. My recommendation: **show workout calories as information only, never add them to the budget**, since our TDEE already accounts for activity level.

## Problem 2 — Rounding at the Wrong Step
### What is it saying?

Our macro calculation has 3 steps: calculate Protein → calculate Fat → calculate Carbs from the remainder. The error is in how we handle the decimal places between steps.

```
Example (incorrect order):
Total Target = 2,995 kcal

Fat Calories = 2,995 × 0.25 = 748.75 kcal
→ We round this to 749 immediately
→ Then subtract: 2,995 - 660 - 749 = 1,586 kcal for Carbs
→ Carbs = 1,586 / 4 = 396.5g → rounds to 397g

✅ Correct order (round only at the very end):
Fat Calories = 748.75 (kept as decimal, not rounded yet)
Carbs Calories = 2,995 - 660 - 748.75 = 1,586.25 kcal
Fat Grams = 748.75 / 9 = 83.19g → rounds to 83g (now round)
Carbs Grams = 1,586.25 / 4 = 396.56g → rounds to 397g (now round)
```

The final numbers look almost the same (1-2 gram difference). But the real risk is that when our Adaptive TDEE engine recalculates targets using stored values from the database, it reuses already-rounded numbers as if they are exact, and those small errors compound over time into something meaningfully wrong.

### Is this important?
**Moderately important.** The current drift is tiny ($1$-$2$ grams). But since the Adaptive TDEE engine in [adaptive-tdee.ts](file:///c:/Fitness_app/lib/engine/adaptive-tdee.ts) will reuse stored profile values, we should fix this before Step 3. It is a 5-line code fix in [tdee.ts](file:///c:/Fitness_app/lib/engine/tdee.ts).

## Problem 4 — No Minimum Floor on Fat Grams
### What is it saying?

Right now, [safety.ts](file:///c:/Fitness_app/lib/engine/safety.ts) only checks one thing: is the total calorie count above the minimum safe floor ($1,200$ for women, $1,500$ for men)?

But it ignores a second danger: **fat grams can fall dangerously low on an aggressive cut.**

```
Example: Lighter woman, LOSE_FAT goal
  TDEE = 1,800 kcal
  Target = 1,800 - 500 = 1,300 kcal

  Fat at 25% = 1,300 × 0.25 = 325 kcal = 36g of fat

  36g/day is below the medical minimum of ~0.5g per kg needed
  for hormone production, vitamin absorption, and joint health.
```

The calorie total ($1,300$) passes the safety floor check. But the fat grams are dangerously low, and our safety module would let this through without a warning.

### Is this important?
**Moderately important.** It won't affect the majority of users (heavier people or those with moderate goals have plenty of room). But it can cause harm for lighter users on aggressive cuts. We should add a fat-gram floor check to [safety.ts](file:///c:/Fitness_app/lib/engine/safety.ts) before Step 3. A simple additional rule: fat grams must never go below $0.5 \times \text{weight in kg}$.

---

## Problem 5 — No Body Composition Adjustment
### What is it saying?

Mifflin-St Jeor knows only: weight, height, age, sex. It does **not** know how much of that weight is muscle vs. fat.

A $75\text{ kg}$ athlete who is $10\%$ body fat has dramatically more metabolically active muscle tissue than a $75\text{ kg}$ person who is $30\%$ body fat. Their real BMR could differ by $200$-$300\text{ kcal/day}$. But both get the same formula output.

There is a more accurate formula (Katch-McArdle) that uses body fat percentage. The reviewer is saying we don't use it, which is fine, but it is a known limitation.

### Is this important?
**No.** The reviewer themselves said: *"this is a known and acceptable simplification."* We would need the user to know their body fat percentage to use Katch-McArdle, which most people don't. More importantly, this is **exactly what the Adaptive TDEE module fixes.** After $14$ days of real weight and calorie logging, [adaptive-tdee.ts](file:///c:/Fitness_app/lib/engine/adaptive-tdee.ts) corrects whatever the formula got wrong by looking at actual results. The formula is just the starting estimate. Real data replaces it over time.
